OpenShift Software Catalog 운영 AI Assistant 앱 개발 기획서

Red Hat OpenShift web console의 Software Catalog에 등록 가능한 운영형 AI 앱을 설계한다. 목표는 단순 챗봇이 아니라, dashboard, context-aware assistant, 운영 지식 RAG, audit/evaluation을 하나의 제품 경험으로 묶는 것이다.

작성일 2026-06-12 KST 대상 OpenShift Web Console 형태 Console Plugin + Operator + Backend 상태 Product Planning Draft

1. 목표, 완료 조건, 검증 방법, 하지 않을 것

구분 내용 측정 방법 현재 gap
목표 OpenShift 운영자가 alert, workload, 로그, 이벤트, GitOps 상태, 사내 runbook을 한 화면에서 해석하고, AI와 함께 안전하게 조치안을 만들 수 있게 한다. 대표 장애 시나리오에서 원인 후보, 근거, 다음 조치까지 도달하는 시간 측정 기존 assistant류 경험은 대화창 중심이며 운영 dashboard와 evidence workflow가 약하다.
완료 조건 Dashboard, context-aware chat, 지식/RAG 관리, 모델/비용 관리, audit/evaluation 화면을 MVP에서 최소 동작시킨다. acceptance criteria pass/fail, e2e demo, RBAC별 화면 확인 대화 UI만 만들면 제품 완성도가 부족하다.
검증 방법 실제 OpenShift alert, pod failure, rollout failure, quota/limit, image pull, monitoring data missing 시나리오를 replay한다. golden scenario, answer citation audit, UI occlusion test, action safety test 모델 답변만 보는 검증으로는 운영 UX를 증명할 수 없다.
하지 않을 것 초기 버전에서 자동 apply/delete/scale 같은 파괴적 조치는 하지 않는다. AI 답변을 무근거로 확정 판정하지 않는다. action policy와 audit log 점검 AI가 운영 권한을 바로 행사하면 보안/책임 경계가 흐려진다.

2. 제품 가설

이 문서는 사용자의 제품 가설을 출발점으로 한다. 현행 Red Hat OpenShift Lightspeed에 대한 세부 평가는 환경과 버전에 따라 달라질 수 있으므로, 아래 항목은 "우리가 만들 앱이 반드시 해결해야 할 문제"로 고정한다.
가설 A. Dashboard 부재

운영자는 질문하기 전 이미 cluster 상태, alert 우선순위, 영향을 받는 workload, 최근 변경점을 보고 싶다. 챗봇만 있으면 문제를 발견하는 surface가 빈약하다.

가설 B. BYOK/외부 지식 주입 의존

사용자가 지식이나 key를 직접 가져오는 방식에만 기대면 RAG 범위가 좁아진다. 공식 문서, 사내 runbook, GitOps repo, ticket/postmortem, live cluster state가 함께 들어와야 한다.

가설 C. 화면을 가리는 UX

운영 assistant가 alert table, log, event, YAML을 가리면 도구가 증거를 해석하는 게 아니라 증거를 숨기는 꼴이 된다.

가설 D. 운영 검증 루프 부족

답변이 맞는지, 어떤 문서를 근거로 했는지, 비용과 latency가 어떤지, 누가 어떤 질문을 했는지 추적되지 않으면 enterprise 운영 제품이 되기 어렵다.

3. 제품 포지셔닝

구분 일반 챗봇 본 제품
첫 화면 질문 입력창 운영 dashboard와 active incident queue
주요 입력 사용자 자연어 자연어 + 현재 console route + selected resource + alert/event/log/YAML + GitOps 변경 이력
RAG 사용자가 넣은 문서 또는 제한된 문서 공식 문서, 사내 runbook, customer docs, cluster state snapshot, incident history, GitOps manifest를 policy 기반으로 통합
UI floating overlay inline dock, resizable drawer, bottom mode, detached window, context chips
답변 문장형 추천 근거, confidence, inspected evidence, next command, rollback path, related docs, risk 표시
운영 통제 제한적 RBAC, data redaction, usage quota, audit log, evaluation dashboard, model fallback

4. 핵심 문제점 정리

4.1 Dashboard 없음

운영자는 챗봇에게 묻기 전에 이미 무엇이 위험한지 알아야 한다. dashboard가 없으면 assistant는 reactive tool에 머물고, 사용자는 alert page, monitoring page, events, logs, workloads를 오가며 증거를 직접 모아야 한다.

문제 사용자 영향 해결 방향
Active alert 우선순위가 assistant 화면과 분리됨 사용자가 먼저 어떤 alert를 물어봐야 할지 판단해야 함 AI Triage Dashboard에서 severity, blast radius, recent change, runbook match를 함께 보여준다.
Cluster health와 지식 health가 분리됨 답변 품질 저하 원인이 cluster data인지 RAG data인지 알기 어려움 Cluster signal freshness와 RAG index freshness를 같은 dashboard에 표시한다.
질문/답변이 운영 backlog로 남지 않음 반복 장애를 학습하거나 개선하기 어렵다 Conversation을 incident note, runbook candidate, evaluation sample로 승격한다.

4.2 BYOK/좁은 RAG 의존

여기서 BYOK는 사용자가 직접 key나 knowledge를 가져와야 확장되는 방식 전체를 넓게 부르는 표현이다. 운영 AI는 단순 문서 검색기가 아니라 "환경별 운영 지식 시스템"이어야 한다.

RAG가 좁으면 답변은 그럴듯해도 실제 cluster, 사내 표준, 고객사 예외, 과거 장애 이력을 놓친다. 운영자가 원하는 것은 일반 OpenShift 설명이 아니라 "우리 cluster에서 지금 무엇을 봐야 하는가"다.

4.3 화면을 가리는 UI

assistant가 alert table의 `심각도`, `합계`, `상태` 컬럼이나 log/YAML을 가리면, 사용자는 챗봇을 열고 닫으며 기억에 의존하게 된다. 운영 도구에서 이것은 polish 문제가 아니라 workflow blocker다. 

원칙:
assistant must not obscure the evidence it is supposed to help interpret.

4.4 Governance와 evaluation 부족

enterprise catalog 앱은 "답변 잘함"만으로 부족하다. 누가 어떤 데이터를 보냈는지, 어떤 모델을 썼는지, 비용과 latency가 얼마인지, 답변이 어떤 근거에 기반했는지 기록되어야 한다.

5. 해결 제품 컨셉

제품명 working title: OpenShift Ops Intelligence Assistant. 한 줄 정의: OpenShift 운영 화면을 가리지 않고, live cluster context와 사내 운영 지식을 결합해 증거 기반 조치안을 제시하는 Software Catalog 앱.
Dashboard-first

질문창보다 운영 현황, 위험 우선순위, 지식 상태, 모델 상태를 먼저 보여준다.

Context-aware

현재 console route, namespace, resource, selected tab, alert filter를 context chip으로 표시하고 답변에 반영한다.

Evidence-first RAG

공식 문서와 사내 문서, live cluster state, GitOps manifest, incident history를 출처별로 분리하고 citation을 강제한다.

Operator-grade governance

RBAC, Secret redaction, audit, quota, model fallback, evaluation set을 기본 기능으로 둔다.

6. 주요 화면 설계

6.1 Operations Dashboard

영역 표시 내용 사용자 행동
Active Incident Queue firing alerts, severity, affected namespaces, last change, related runbook match Ask with context, open runbook, create incident note
Cluster Health Map nodes, operators, workloads, routes, storage, monitoring freshness drill down, compare recent state, attach snapshot
Risk Radar crashloop, pending pods, quota pressure, image pull, certificate, update risk generate triage checklist
Knowledge Health indexed docs, stale docs, failed connectors, low citation coverage re-index, assign owner, add source
Model and Cost provider status, token use, latency, failure rate, fallback count set quota, change routing policy

6.2 Assistant Panel

기능 설계 Pass 기준
Inline dock 오른쪽 overlay 대신 page layout을 나누는 dock mode 제공 Alerts table의 핵심 컬럼이 계속 보인다.
Resizable 좌측 splitter handle로 width 조절, 마지막 크기 저장 최소/최대 폭과 keyboard 조작이 동작한다.
Bottom mode wide table, log, event 화면에서는 하단 dock 선택 오른쪽 table 컬럼이 가려지지 않는다.
Detached window 새 창으로 분리하되 session_id로 console context 동기화 namespace/resource/filter context가 분리창에서도 유지된다.
Peek mode 누르고 있는 동안 투명도 조절 또는 click-through 접근성 경고와 함께 임시 확인 용도로만 제공한다.
Context chips 현재 `cluster`, `namespace`, `kind/name`, `tab`, `filter`를 칩으로 표시 사용자가 AI가 무엇을 보고 답하는지 확인할 수 있다.

7. RAG와 지식 운영 설계

7.1 Knowledge Source 계층

계층 예시 처리 원칙
Official OpenShift docs, Red Hat KB, product release notes 버전 metadata와 함께 citation 우선순위 높게 유지
Customer 사내 runbook, 장애 대응 절차, 고객사 표준 운영 문서 tenant/project/owner/expiry metadata 필수
Live Cluster alerts, events, pods, operators, routes, nodes, metrics snapshot read-only, TTL, RBAC filter, Secret redaction
Change Source GitOps repo, Argo CD app, deployment rollout, image tag change 장애 전후 변경점 correlation에 사용
Incident Memory past Q&A, resolved incidents, postmortems, ticket comments 검증된 항목만 reusable knowledge로 승격

7.2 Retrieval Pipeline

question + console_context
  -> intent classification
  -> RBAC scoped cluster context fetch
  -> hybrid retrieval: BM25 + vector + metadata filter
  -> rerank by product version, namespace, recency, source trust
  -> evidence pack
  -> answer generation with citations
  -> safety/risk check
  -> audit + evaluation sample candidate

7.3 RAG Health Dashboard

Coverage질문군별 citation 가능 비율
Freshness문서와 cluster snapshot 갱신 시간
Conflict공식 문서와 사내 문서 충돌 건수
Groundedness답변 문장별 근거 매칭률

10. Safety, RBAC, Audit

원칙 구현 검증
사용자가 볼 수 없는 리소스는 AI도 볼 수 없다 console user token 또는 impersonation 기반 RBAC scoped fetch 권한 없는 namespace 질의 시 context 미첨부
Secret은 기본적으로 답변에 포함하지 않는다 Secret kind 차단, env/token/password pattern redaction redaction unit test, prompt snapshot audit
초기 조치는 read-only 명령어/patch 생성만 제공, apply는 approval workflow 이후 destructive verb 차단 테스트
모든 답변은 추적 가능해야 한다 request_id, user, sources, model, token, latency, context hash 저장 audit dashboard와 export 확인
근거 없는 확정 판정 금지 confidence와 missing evidence를 답변 template에 포함 golden scenario에서 unsupported claim 탐지

11. Acceptance Criteria

영역 Pass 기준 측정 방법 Evidence
Dashboard firing alert 5개 이상 환경에서 priority queue와 affected resources가 표시된다. fixture cluster 또는 mock API replay screenshot, API response, test log
UI 가림 방지 Alerts page에서 assistant open 상태여도 severity/count/status 확인 가능 desktop/mobile screenshot diff Playwright screenshot, occlusion checklist
Context sync 현재 route, namespace, resource, selected tab이 assistant context chips에 표시된다. console navigation e2e payload snapshot, UI screenshot
RAG citation 운영 답변의 핵심 주장 80% 이상이 source citation과 연결된다. answer audit script citation report
RBAC 권한 없는 namespace 리소스가 retrieval/context에 포함되지 않는다. two-user permission test audit log, denied context report
Action safety delete/apply/scale 같은 조치는 기본 상태에서 실행되지 않고 plan으로만 생성된다. policy test blocked action audit
Evaluation golden scenario 20개 이상을 nightly로 돌리고 regression을 표시한다. CI job evaluation dashboard

15. 결론

이 앱은 Lightspeed 같은 assistant를 따라 만드는 제품이 아니라, Lightspeed류 assistant가 놓치기 쉬운 운영 제품의 빈틈을 정면으로 보완하는 제품이어야 한다.

핵심 차별점은 세 가지다. 첫째, 질문창보다 dashboard가 먼저다. 둘째, RAG는 문서 검색이 아니라 운영 지식 공급망이다. 셋째, assistant UI는 사용자가 해석해야 할 증거를 절대 가리면 안 된다.
