그대로 읽는 대본

0:00-1:00

안녕하세요. 오늘은 OCP 내부에서 운영 콘솔 기능을 어디까지 확장할 수 있는지 확인한 PoC 결과를 공유드리겠습니다.

이번 작업의 핵심은 단순히 별도 웹 대시보드를 만든 것이 아닙니다. OpenShift가 공식적으로 제공하는 확장 경로 안에서, 우리가 Cywell OpsLens라는 운영 콘솔 모드를 어디까지 붙일 수 있는지 검증한 것입니다.

정확히 말하면 Cywell OpsLens는 OpenShift Console을 대체하는 제품이 아니라, OCP 웹콘솔에 운영 기능이 추가된 맞춤형 콘솔 확장 모드입니다.

1:00-3:00

구성 방식은 이렇게 잡았습니다. 먼저 OpenShift Console은 그대로 유지합니다. 그 위에 Cywell OpsLens 진입점을 추가합니다.

설치는 Software Catalog 또는 OperatorHub를 통해 진행합니다. OpenShift에서는 이런 설치형 소프트웨어를 Operator라고 부르는 것이 정확합니다.

Operator가 설치되면 OpsLensInstallation이라는 Custom Resource를 기준으로 API, dashboard, ConsolePlugin 리소스를 적용합니다.

ConsolePlugin은 OpenShift가 공식 지원하는 콘솔 확장 방식입니다. 이것을 통해 좌측 메뉴에 Cywell OpsLens를 추가하고, 사용자가 클릭하면 OCP 콘솔 안에서 full-page OpsLens 운영 화면으로 진입합니다.

즉 iframe으로 억지로 넣거나, DOM을 직접 바꾸거나, OpenShift 콘솔을 갈아끼우는 방식이 아닙니다. 공식 ConsolePlugin, OperatorHub, OLM 흐름을 따르는 구조입니다.

3:00-5:00

이번에 확인한 첫 번째 팩트는, OCP 웹콘솔 기능을 기능 단위로 1:1 매핑한 커스텀 운영 모드 개발이 가능하다는 점입니다.

OpenShift Dynamic Plugin은 custom page, navigation item, resource tab, action 같은 확장 지점을 제공합니다. 그래서 기존 OCP 콘솔 기능을 OpsLens 안에서 매핑하고, 그 위에 더 보기 쉬운 운영 화면, 리소스 조회, Assistant, runbook 기반 분석 기능을 붙일 수 있습니다.

두 번째는 Operator 활용입니다. OCP 안에서 제공되는 인앱 소프트웨어는 공식적으로 Operator, Software Catalog, OperatorHub 흐름으로 설명하는 것이 맞습니다. Cywell OpsLens 자체도 Operator로 패키징해서 설치형 제품으로 제공할 수 있습니다.

세 번째는 Lightspeed 연동입니다. OpenShift Lightspeed를 대체하는 것이 아니라, Lightspeed를 온전히 쓰면서 OpsLens의 고객사 RAG, runbook, 장애 분석 도구를 더하는 구조가 가능합니다.

5:00-7:00

Lightspeed 쪽은 세 가지 경로로 정리했습니다. 첫 번째는 Lightspeed REST API입니다. OpsLens Assistant가 Lightspeed의 공식 OpenShift 답변을 호출하고, 여기에 현재 콘솔 context와 OpsLens RAG 근거를 결합할 수 있습니다.

두 번째는 BYO Knowledge입니다. 고객사 Markdown 문서나 runbook을 RAG 지식으로 구성해서 Lightspeed 답변 근거로 사용할 수 있습니다.

세 번째는 custom MCP server입니다. OpsLens를 MCP server로 붙이면 Lightspeed가 고객사 운영 도구, 장애 분석, read-only OCP 조회, 승인형 작업 계획을 호출할 수 있습니다.

중요한 것은 native Lightspeed drawer를 갈아끼우겠다는 것이 아닙니다. native Lightspeed는 그대로 쓰고, OpsLens를 plus alpha로 붙이는 것이 공식 근거상 안전한 방향입니다.

7:00-8:30

또 하나 중요한 질문은 쓰기 작업이 가능한지입니다. 결론부터 말하면 가능합니다. 다만 챗봇이 직접 클러스터를 마음대로 수정하는 구조는 아닙니다.

정확한 구조는 Assistant가 진단을 하고, ActionPlan을 만들고, 각 action에 대해 RBAC preflight를 수행한 뒤, human approval을 받고, 승인된 allowlist action만 OpsLens API나 Operator가 적용하는 방식입니다.

Dev 0.1.6에서는 이 방향을 Agentic Operations로 정리했습니다. read-only 기본값은 유지하되, 승인된 안전 작업은 실행, 검증, audit trail까지 남기는 구조로 확장하는 단계입니다.

8:30-9:30

프로젝트 진행 상태도 단계별로 정리했습니다. Dev 0.1.1에서는 CRC OperatorHub 설치 경로를 확인했습니다. Dev 0.1.2에서는 KOMSCO UI와 고객용 화면을 정리했습니다. Dev 0.1.3에서는 공식 ConsolePlugin 방향을 고정했습니다.

Dev 0.1.4에서는 OpenShift 콘솔 좌측 메뉴에서 Cywell OpsLens가 실행되고 API 연결됨 상태까지 확인했습니다. Dev 0.1.5에서는 발표 가능한 운영 대시보드와 movable KOMSCO AI Assistant를 만들었습니다.

최신 Dev 0.1.6에서는 live console mod, Lightspeed console proxy, CRC Lightspeed network policy 근거를 추가했고, ActionPlan, RBAC, approval, audit 기반 쓰기 작업 방향을 고정했습니다.

9:30-10:00

정리하면, Cywell OpsLens의 가치는 OpenShift를 바꾸는 별도 콘솔을 만드는 것이 아닙니다.

OpenShift Console과 Lightspeed를 공식 확장 경로로 그대로 활용하면서, 고객사 고유 운영 지식, RAG, runbook, 운영 대시보드, Assistant, 승인형 자동화까지 결합하는 맞춤형 운영 콘솔 모드를 제공하는 것입니다.

오늘 발표의 결론은 구현 완료 선언이 아니라, 공식 근거와 실제 PoC로 제품 방향이 타당하다는 것을 확인했다는 점입니다. 이상입니다.

예상 질문 답변

질문 답변
OpenShift 콘솔을 스킨처럼 갈아끼우는 건가요? 아닙니다. 공식 ConsolePlugin으로 메뉴와 full-page 화면을 추가하는 운영 콘솔 확장 모드입니다.
Operator라는 표현이 맞나요? 네. OCP Software Catalog / OperatorHub에서 설치되는 인앱 소프트웨어는 Operator로 표현하는 것이 정확합니다.
Lightspeed를 대체하나요? 아닙니다. native Lightspeed는 유지하고 REST API, BYO Knowledge, MCP로 고객사 지식과 OpsLens 도구를 붙이는 구조입니다.
쓰기 작업도 가능한가요? 가능합니다. 다만 RBAC, human approval, allowlist, rollback, audit를 통과한 승인형 자동화로 설계해야 합니다.
오늘 production ready인가요? 아닙니다. 오늘은 공식 가능 범위와 PoC 진행 상태를 확인한 발표입니다. production runtime, 고객 데이터, OLSConfig patch, 회사 OCP mutation은 별도 승인 단계입니다.

마무리 한 문장

Cywell OpsLens는 OpenShift와 Lightspeed를 대체하는 것이 아니라, 공식 확장 경로 안에서 고객사 고유 운영 지식과 승인형 자동화를 더하는 OCP 맞춤형 운영 콘솔 모드입니다.
