Cywell OpsLens CRC Registry Bootstrap Runbook

작성일: 2026-06-17 · 대상: 개인 MacBook CRC OpenShift · 범위: 내부 registry 이미지 적재와 트러블슈팅

현재 결론: MacBook CRC OpenShift 내부 registry에 cywell-opslens-api:verify, cywell-opslens-dashboard:verify, cywell-opslens-operator:verify 이미지를 정상 등록했다.

1. 목표

회사 공용 OpenShift가 아니라 개인 MacBook의 CRC OpenShift를 Cywell OpsLens 개발/검증 대상 클러스터로 삼고, 회사 노트북에서 빌드한 verify 이미지를 MacBook CRC의 OpenShift internal registry에 올린다.

  • 회사 공용 OCP 리소스와 충돌하지 않는 개인 실험 환경을 확보한다.
  • API, Dashboard, Operator 이미지를 CRC 내부에서 pull 가능한 상태로 만든다.
  • 이후 Operator/Deployment/Console Plugin 설치 검증으로 넘어갈 수 있는 기반을 만든다.

2. 완료 조건

항목 완료 기준 검증 방법 상태
CRC 클러스터 상태 OpenShift가 Running이고 ClusterVersion/ClusterOperator가 정상 crc status, oc get clusterversion, oc get co PASSRunning 확인
대상 project cywell-opslens project 접근 가능 oc whoami && oc project -q PASSkubeadmin / cywell-opslens
Registry 인증 OpenShift image registry에 Docker credential 저장 oc registry login --to=/tmp/cywell-docker-config/config.json PASScredential 저장
Registry TLS Docker Desktop이 CRC route CA를 신뢰 route CA를 Mac System Keychain에 trustRoot 등록 후 Docker Desktop 재시작 PASSpush TLS 통과
ImageStream API/Dashboard/Operator ImageStream 존재 oc create imagestream ... PASSmanifest 등록 성공
이미지 태그 세 이미지가 :verify tag로 조회됨 oc get istag -n cywell-opslens PASS세 tag 조회됨

3. 보안/기록 원칙

  • 이 문서에는 토큰, 비밀번호, 정확한 사설 IP, 정확한 registry route host를 고정 기록하지 않는다.
  • 명령은 <CRC_REGISTRY_ROUTE>, <PROJECT> 같은 변수로 표현한다.
  • 회사 공용 OCP에는 적용하지 않는다. 이 runbook은 개인 MacBook CRC 전용이다.
  • 초기 단계에서는 이미지 적재와 read-only 확인 중심으로 진행하고, cluster mutation은 명시된 CRC project 내부로 제한한다.

4. 권장 변수

아래 변수는 실제 터미널에서만 설정한다. 문서나 Git에는 실제 값을 남기지 않는다.

PROJECT=cywell-opslens
REGISTRY_ROUTE=<CRC image registry route host>
DOCKER_CONFIG_DIR=/tmp/cywell-docker-config

5. 성공한 최종 절차

5.1 CRC와 OCP 상태 확인

crc status
oc whoami && oc project -q
oc get clusterversion
oc get co

확인 기준은 OpenShift Running, ClusterVersion Available True, ClusterOperator Available True/Progressing False/Degraded False다.

5.2 Local image 존재 확인

docker images --format '{{.Repository}}:{{.Tag}}' | grep -E '^cywell/opslens-(api|dashboard|operator):verify$'

Mac Docker Desktop에 다음 이미지가 있어야 했다.

  • cywell/opslens-api:verify
  • cywell/opslens-dashboard:verify
  • cywell/opslens-operator:verify

5.3 Registry route와 권한 확인

oc get route default-route -n openshift-image-registry
oc auth can-i update imagestreams/layers -n "$PROJECT"

can-i 결과가 yes면 push 권한은 있는 것이다.

5.4 Docker credential 저장

mkdir -p "$DOCKER_CONFIG_DIR"
oc registry login \
  --registry="$REGISTRY_ROUTE" \
  --to="$DOCKER_CONFIG_DIR/config.json" \
  --skip-check

5.5 CRC registry route CA 추출

mkdir -p "$HOME/.docker/certs.d/$REGISTRY_ROUTE" "$HOME/.docker/certs.d/$REGISTRY_ROUTE:443"
oc get secret router-ca -n openshift-ingress-operator \
  -o jsonpath='{.data.tls\.crt}' \
  | base64 -d > "$HOME/.docker/certs.d/$REGISTRY_ROUTE/ca.crt"
cp "$HOME/.docker/certs.d/$REGISTRY_ROUTE/ca.crt" \
  "$HOME/.docker/certs.d/$REGISTRY_ROUTE:443/ca.crt"

5.6 CA가 맞는지 검증

openssl s_client -connect "$REGISTRY_ROUTE:443" -servername "$REGISTRY_ROUTE" </dev/null 2>/dev/null \
  | openssl x509 -noout -subject -issuer
openssl x509 -in "$HOME/.docker/certs.d/$REGISTRY_ROUTE/ca.crt" -noout -subject
curl --connect-timeout 5 -sS \
  --cacert "$HOME/.docker/certs.d/$REGISTRY_ROUTE/ca.crt" \
  "https://$REGISTRY_ROUTE/v2/"

curl 결과가 UNAUTHORIZED면 TLS 검증과 registry 도달은 성공이다. 인증이 필요한 registry API라는 뜻이다.

5.7 Mac System Keychain에 CA 신뢰 등록

이 단계는 SSH 터미널에서 실패한다. MacBook 화면의 로컬 Terminal 앱에서 직접 실행해야 한다. 
sudo security add-trusted-cert \
  -d -r trustRoot \
  -k /Library/Keychains/System.keychain \
  "$HOME/.docker/certs.d/$REGISTRY_ROUTE/ca.crt"

성공 후 Mac Docker Desktop을 재시작한다.

docker info >/dev/null && echo DOCKER_READY

5.8 ImageStream 선생성

oc create imagestream cywell-opslens-api -n "$PROJECT"
oc create imagestream cywell-opslens-dashboard -n "$PROJECT"
oc create imagestream cywell-opslens-operator -n "$PROJECT"

5.9 이미지 tag/push

docker tag cywell/opslens-api:verify "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-api:verify"
DOCKER_CONFIG="$DOCKER_CONFIG_DIR" docker push "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-api:verify"

docker tag cywell/opslens-dashboard:verify "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-dashboard:verify"
DOCKER_CONFIG="$DOCKER_CONFIG_DIR" docker push "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-dashboard:verify"

docker tag cywell/opslens-operator:verify "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-operator:verify"
DOCKER_CONFIG="$DOCKER_CONFIG_DIR" docker push "$REGISTRY_ROUTE/$PROJECT/cywell-opslens-operator:verify"

5.10 최종 검증

oc get istag -n "$PROJECT"

다음 세 tag가 보여야 한다.

  • cywell-opslens-api:verify
  • cywell-opslens-dashboard:verify
  • cywell-opslens-operator:verify

6. 트러블슈팅 기록

증상 당시 판단 확정 원인 해결
docker push가 registry route에서 x509: certificate signed by unknown authority로 실패 권한/네트워크/CA 후보를 분리해야 했음 Mac Docker Desktop이 CRC registry route CA를 신뢰하지 못함 router CA 추출 → Mac System Keychain에 trustRoot 등록 → Docker Desktop 재시작
curl -vk https://127.0.0.1:5000/v2/가 401 Unauthorized 반환 실패처럼 보였지만 좋은 신호 registry API에 도달했고 인증만 필요한 상태 브라우저 접근 거부는 정상으로 판단, Docker 인증/TLS 문제로 범위 축소
0.0.0.0:5000 bind: address already in use 기존 port-forward 잔류 의심 Mac의 다른 프로세스가 5000 포트를 점유 lsof -nP -iTCP:5000 -sTCP:LISTEN으로 확인 후 외부 포트를 5001로 변경
host.docker.internal:5001 push가 인증서 이름 불일치로 실패 Docker Desktop 내부 엔진이 Mac host로 접근은 가능 registry 인증서 CN/SAN이 host.docker.internal용이 아님 port-forward 우회 대신 registry route CA 신뢰 방식으로 전환
sudo security add-trusted-cert가 SSH에서 no user interaction was possible로 실패 sudo 비밀번호 문제가 아니라 macOS 보안 정책 문제 System Keychain trust 변경은 GUI 사용자 상호작용이 필요 MacBook 로컬 Terminal 앱에서 직접 실행
docker run alpine이 Keychain 접근 오류로 실패 OpenShift 문제가 아니라 Docker Hub pull 문제 SSH 세션에서 Docker credential helper가 Mac Keychain을 열 수 없었음 외부 pull이 필요 없는 로컬 Cywell 이미지로 검증 경로 변경
push 마지막에 500 Internal Server Error 처음에는 Mac/CRC 디스크 98% 때문에 공간 부족도 의심 registry 로그상 imagestreams.image.openshift.io "...api" not found ImageStream을 먼저 생성한 뒤 push 재시도
oc get istag ...가 NotFound 레이어는 올라갔지만 tag/manifest는 실패한 상태 ImageStream 부재로 manifest/tag 등록이 되지 않았음 oc create imagestream 후 push 재실행
image-pruner pod가 ContainerCreating에 오래 머무름 디스크 부족 가능성도 같이 의심 event상 직접 원인은 CNI/Multus Unauthorized였고, push 실패의 직접 원인은 아니었음 registry 로그에서 실제 status=500 원인을 추적하여 ImageStream 문제로 분리

7. 왜 이 과정이 중요했나

이번 문제는 단순히 명령을 여러 번 재시도해서 해결한 것이 아니라, 실패 지점을 계층별로 분리해서 해결했다.

  1. 권한: oc auth can-i, oc registry login으로 확인
  2. 네트워크: curl /v2/의 401 응답으로 확인
  3. TLS: registry issuer와 router CA subject 비교, curl --cacert로 확인
  4. Docker Desktop 신뢰 저장소: Mac System Keychain 등록으로 해결
  5. OpenShift registry 객체 모델: ImageStream 선생성으로 manifest/tag 등록 해결

이 분리 덕분에 “Docker가 이상하다”, “CRC가 이상하다” 같은 막연한 결론 대신, Mac Docker Desktop trust path와 OpenShift ImageStream 선행 조건이라는 실제 원인을 남길 수 있었다.

8. 현재 남은 일

다음 단계 설명 주의점
API/Dashboard 배포 내부 registry의 :verify 이미지를 사용해 Deployment, Service, 필요 시 Route 생성 우선 CRC namespace 내부에서만 수행
Operator 배포 경로 선택 직접 operator Deployment로 갈지, OLM bundle/catalog 경로로 갈지 결정 OLM 정식 경로에는 bundle/catalog 이미지가 추가로 필요
Lightspeed 연동 read-only readiness와 patch preview를 먼저 확인 OLSConfig 실제 patch는 명시 승인 전 금지
Console Plugin 등록 Cywell OpsLens를 OCP Console 경험에 붙이는 단계 먼저 dashboard/API가 CRC에서 정상 응답해야 함
디스크 정리 Mac host와 CRC VM 디스크 여유 확보 이미지 push/LLM/vLLM 실험 전에 여유 공간 필요

9. 하지 않을 것

  • 회사 공용 OCP에는 이 실험 이미지를 push하거나 설정을 변경하지 않는다.
  • 토큰, kubeconfig, registry credential, exact private route/IP를 Git에 남기지 않는다.
  • Lightspeed/OCP 리소스에 자동 apply/delete/scale을 하지 않는다.
  • ImageStream/Deployment 문제를 디스크 문제로 단정하지 않는다. 로그와 이벤트로 원인을 분리한다.
  • OLM 설치를 우회해 성공한 척하지 않는다. 직접 배포와 OLM 설치는 별도 경로로 기록한다.

10. 빠른 재현 체크리스트

  1. Mac CRC가 Running인지 확인한다.
  2. oc whoami와 project가 맞는지 확인한다.
  3. Mac Docker에 verify 이미지가 있는지 확인한다.
  4. registry route credential을 /tmp/cywell-docker-config에 저장한다.
  5. router CA를 추출하고 Mac System Keychain에 trustRoot로 등록한다.
  6. Docker Desktop을 재시작한다.
  7. ImageStream을 먼저 만든다.
  8. 이미지를 tag/push한다.
  9. oc get istag로 tag/digest를 확인한다.
운영 판단: 이번 산출물의 핵심은 “성공 명령 모음”이 아니라 실패를 계층별로 분해한 기록이다. 다음 설치 단계에서도 같은 방식으로 권한, 네트워크, TLS, OpenShift 객체 계약, 애플리케이션 readiness를 분리해서 검증한다.
