Cywell OpsLens Mac CRC Daily Reconnect Checklist

작성일: 2026-06-17 · 목적: 퇴근 후 MacBook 이동/재연결 뒤 회사 노트북에서 같은 개발 연결상태를 재현한다.

현재 성공 상태: 회사 노트북에서 개발하고, MacBook의 CRC OpenShift를 target OCP로 사용하며, SSH tunnel을 통해 Mac CRC의 OpenShift Lightspeed port-forward까지 접근 가능하다.

1. 현재 연결 구조

구간 역할 현재 판단
회사 노트북 개발 repo, .env, npm 검증 명령 실행 위치 ACTIVE주 개발 워크스테이션
MacBook CRC OpenShift 서버, Docker Desktop, FortiClient, Lightspeed port-forward 실행 위치 TARGET개인 OCP lab
CRC OpenShift cywell-opslens namespace와 OpenShift Lightspeed 서비스 제공 RUNNINGv4.21 계열 확인
Windows SSH tunnel 회사 노트북 local port를 MacBook local Lightspeed port-forward로 연결 WORKING18443 -> Mac 8443

2. 내일 아침 복구 순서

  1. MacBook을 켜고 네트워크에 연결한다.
  2. MacBook에서 FortiClient를 연결한다. 회사 LLM/사내 endpoint가 필요하면 Mac 쪽 VPN이 중요하다.
  3. 회사 노트북에서 MacBook SSH 접속이 되는지 확인한다.
  4. MacBook CRC가 Running인지 확인한다.
  5. MacBook SSH 터미널 하나에서 Lightspeed port-forward를 켠다.
  6. 회사 노트북 PowerShell 하나에서 SSH local tunnel을 켠다.
  7. 회사 노트북 repo PowerShell에서 Lightspeed base URL을 18443으로 세션 override한다.
  8. HTTP 404 응답으로 연결 경로가 살아 있는지 확인한다.

3. 터미널별로 해야 할 일

3.1 MacBook SSH 터미널 A: CRC 상태 확인

crc status
oc whoami && oc project -q
oc get clusterversion
oc get co

정상 기준은 CRC VM Running, OpenShift Running, oc whoami 가능, cluster progressing false다.

3.2 MacBook SSH 터미널 B: Lightspeed port-forward 유지

oc -n openshift-lightspeed port-forward svc/lightspeed-app-server 8443:8443

이 창은 닫지 않는다. 정상 출력은 아래와 비슷하다.

Forwarding from 127.0.0.1:8443 -> 8443
Forwarding from [::1]:8443 -> 8443

3.3 MacBook SSH 터미널 C: Mac 내부에서 Lightspeed 확인

curl -k -sS --connect-timeout 5 -o /dev/null \
  -w 'HTTP=%{http_code} TIME=%{time_total}\n' \
  https://127.0.0.1:8443

HTTP=404면 정상이다. 루트 path가 없다는 뜻이지, 연결 실패가 아니다.

3.4 회사 노트북 PowerShell A: SSH local tunnel 유지

ssh -N -L 18443:127.0.0.1:8443 souluk@kugnus-McBook

비밀번호 입력 후 아무 출력 없이 멈춰 있으면 정상이다. 이 창도 닫지 않는다.

3.5 회사 노트북 repo PowerShell B: 세션 override

cd C:\Users\soulu\cywell\Kugnus-Ops-Lens
$env:OPENSHIFT_LIGHTSPEED_BASE_URL="https://127.0.0.1:18443"
이 값은 현재 PowerShell 창에서만 유효하다. 새 창을 열면 다시 설정해야 한다. .env 파일을 바꾸지 않는 것을 권장한다.

3.6 회사 노트북 repo PowerShell B: 터널 경유 Lightspeed 확인

$envs=@{}; Get-Content .env | % {
  if ($_ -match '^\s*([^#][A-Za-z_][A-Za-z0-9_]*)\s*=\s*(.*)$') {
    $envs[$matches[1]]=$matches[2].Trim('"')
  }
}; $url=$envs.OPENSHIFT_LIGHTSPEED_BASE_URL -replace '127\.0\.0\.1:8443','127.0.0.1:18443'; curl.exe -k -sS --connect-timeout 5 -o NUL -w "HTTP=%{http_code} TIME=%{time_total}`n" -H "Authorization: Bearer $($envs.OPENSHIFT_LIGHTSPEED_API_TOKEN)" "$url"

HTTP=404면 회사 노트북에서 Mac CRC Lightspeed까지 경로가 살아 있는 것이다.

4. 현재 이미지/registry 상태

Mac CRC 내부 registry에는 아래 verify 이미지가 등록되어 있다.

  • cywell-opslens-api:verify
  • cywell-opslens-dashboard:verify
  • cywell-opslens-operator:verify
oc get istag -n cywell-opslens

세 tag가 보이면 이미지 적재 상태는 유지되고 있다. Docker Desktop 재시작이나 Mac 이동만으로 이 tag가 사라지지는 않는다.

5. 헷갈리기 쉬운 포인트

상황 의미 조치
회사 노트북에서 https://127.0.0.1:8443 실패 회사 노트북 자기 자신의 8443을 본 것이다. Mac port-forward가 아니다. Windows SSH tunnel을 켜고 18443으로 접근한다.
Mac에서 https://127.0.0.1:8443이 HTTP=404 성공이다. 서버에는 닿았고 루트 path가 없을 뿐이다. 그대로 진행한다.
ssh -L 8443:...가 bind permission denied 회사 노트북의 8443 포트를 못 잡은 것이다. 회사 노트북 local port를 18443으로 쓴다.
lsof -nP -iTCP:8443 -sTCP:LISTEN 출력 없음 Mac의 Lightspeed port-forward가 꺼져 있다. Mac SSH에서 port-forward를 다시 켠다.
PowerShell override 후 새 창에서 다시 안 됨 $env:OPENSHIFT_LIGHTSPEED_BASE_URL은 창 단위 설정이다. 새 창마다 override 명령을 다시 실행한다.
FortiClient 연결 없이 registry/CRC는 됨 registry push와 CRC 조작은 Mac 내부 작업이라 VPN이 필요 없을 수 있다. 회사 LLM/사내 endpoint 검증 전에는 Mac FortiClient를 연결한다.

6. 내일 바로 이어갈 작업

  1. 이 문서의 복구 체크리스트대로 터널을 재현한다.
  2. 회사 노트북에서 Lightspeed base URL override가 살아 있는지 확인한다.
  3. CRC의 cywell-opslens namespace에 API/Dashboard Deployment를 만든다.
  4. Dashboard Route를 열어 실제 화면을 확인한다.
  5. 그 다음 Operator direct deploy 또는 OLM bundle/catalog 경로를 선택한다.

7. 하지 않을 것

  • 회사 공용 OCP에는 연결하지 않는다.
  • .env의 토큰이나 URL 값을 문서에 박지 않는다.
  • Mac에 repo나 secret 파일을 굳이 복사하지 않는다.
  • Lightspeed OLSConfig patch/apply는 배포와 readiness 확인 전에는 하지 않는다.
운영 판단: 내일의 목표는 연결 자체가 아니라, 이 연결을 기반으로 Cywell OpsLens API와 Dashboard를 CRC 안에서 실제로 띄우는 것이다.
